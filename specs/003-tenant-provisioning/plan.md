# Implementation Plan: Tenant Provisioning

**Branch**: `019-tenant-provisioning` | **Date**: 2026-05-28 | **Spec**: [spec.md](spec.md)

**Input**: [specs/003-tenant-provisioning/spec.md](spec.md)

---

## Summary

Implement the platform operator control plane: `tenant_manager`-only CRUD routes under `/platform/tenants`, suspension/reactivation lifecycle, async deletion trigger, aggregate usage summary, paginated audit log, and suspension enforcement in `require_tenant_admin`. All DB models and migrations already exist from spec 001. This spec fills in the repository, service, route, schema, and test layers.

---

## Technical Context

**Language/Version**: Python 3.12

**Primary Dependencies**: FastAPI, Pydantic v2, SQLAlchemy 2.x async, asyncpg, structlog, pytest, httpx

**Storage**: PostgreSQL 16 (tenants, users, audit_logs, cost_events tables — all exist in migration 0001)

**Testing**: pytest with in-process async test client; existing `tests/` structure

**Target Platform**: Linux server (Docker Compose)

**Project Type**: Web service (FastAPI)

**Performance Goals**: Tenant create + invite-admin round trip < 3 s (SC-001)

**Constraints**: No new migrations needed. Tenant manager must never read content tables. Suspension takes effect within one request (SC-005). Audit writes are fire-and-forget (non-blocking).

**Scale/Scope**: Single-region, Week 8 scope. One seeded `tenant_manager` user. No email invite flow.

### Phase E: Dependency Update

**`dependencies.py`** — `require_tenant_admin`:

After confirming `user.role == tenant_admin`, load the tenant row and check:
```python
if tenant.status == TenantStatus.suspended:
    raise HTTPException(403, detail="Tenant is suspended", headers={"X-Error-Code": "tenant_suspended"})
if tenant.status in (TenantStatus.deleting, TenantStatus.deleted):
    raise HTTPException(403, detail="Tenant not active")
```

This is one extra DB lookup per request on tenant-admin routes — acceptable cost for SC-005 compliance.

### Phase F: Tests

**`tests/test_tenant_provisioning.py`** — covers all spec acceptance scenarios:

- Story 1: create tenant (201), wrong role (403), duplicate slug (409), audit log entry written
- Story 2: invite-admin (201), invited user can login + access `/tenant/config`, wrong role (403)
- Story 3: suspend (200), suspended user gets 403 `tenant_suspended`, reactivate restores access
- Story 4: list tenants excludes content, usage-summary returns aggregates not content, wrong role (403)
- Story 5: delete sets status=deleting, audit event written, 404 on deleted tenant routes
- Edge cases: invite on suspended tenant (422), duplicate email (409), idempotent suspend, invalid slug (422), double-delete (409/200)

Use the existing async test pattern from `tests/test_auth.py`.
